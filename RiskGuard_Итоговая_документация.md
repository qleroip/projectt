# RiskGuard

Итоговая документация проекта

Версия: 1.0  
Дата: 05.05.2026  
Тип проекта: веб-приложение, desktop-приложение и APK-клиент для управления операционными рисками

## 1. Назначение проекта

RiskGuard - система управления операционными рисками предприятия. Проект помогает фиксировать инциденты, создавать риски, назначать экспертов, получать экспертные оценки, формировать меры минимизации и контролировать состояние рисков через веб-панель.

Система разделена на три пользовательских клиента:

- Web-панель для администратора и риск-менеджера.
- Desktop-приложение для эксперта.
- APK-приложение для работника.

Основная идея проекта: работник сообщает об инциденте, риск-менеджер проверяет инцидент и при необходимости создает риск, эксперт оценивает риск, а риск-менеджер управляет статусами и мерами минимизации.

## 2. Цели и задачи

Цели проекта:

- Снизить вероятность потери информации об инцидентах.
- Разделить обязанности между работником, риск-менеджером, экспертом и администратором.
- Сделать понятный процесс обработки риска от инцидента до завершения.
- Реализовать API для связи web, desktop и APK.
- Продемонстрировать архитектурные требования: CQRS, BFF, domain events, очередь событий, eventual consistency, метрики и документацию.

Задачи системы:

- Регистрация и одобрение пользователей администратором.
- Прием инцидентов от работников.
- Создание риска из инцидента.
- Создание риска с мерами минимизации.
- Назначение эксперта на риск.
- Отправка риска в работу только после назначения эксперта.
- Экспертная оценка риска по вероятности и влиянию.
- Просмотр журнала рисков, аудита, фоновых задач и отчетов.
- Экспорт отчета.
- Разделение API под разные клиенты.

## 3. Роли пользователей

### 3.1 Администратор

Администратор отвечает за управление учетными записями. Он одобряет заявки на регистрацию, может видеть пользователей и имеет расширенный доступ для демонстрации и обслуживания системы.

Основные действия администратора:

- Одобрение новых пользователей.
- Управление доступом.
- Просмотр веб-панели.
- Доступ к административным функциям.

### 3.2 Риск-менеджер

Риск-менеджер работает в web-версии. Он является основным пользователем веб-панели.

Основные действия риск-менеджера:

- Просмотр журнала рисков.
- Создание риска и мер минимизации.
- Создание риска из инцидента работника.
- Назначение эксперта.
- Перевод риска в статус "В работе".
- Завершение или отмена риска.
- Просмотр фоновых задач и журнала аудита.
- Экспорт отчета.

### 3.3 Эксперт

Эксперт работает в desktop-приложении. Он не управляет всеми рисками, а видит только назначенные ему риски.

Основные действия эксперта:

- Вход в desktop-приложение.
- Просмотр назначенных рисков.
- Открытие карточки риска.
- Выставление вероятности от 1 до 5.
- Выставление влияния от 1 до 5.
- Написание экспертной рекомендации.
- Отправка оценки.
- Просмотр своих отправленных оценок.

### 3.4 Работник

Работник работает в APK-приложении. Он не создает риск напрямую, а отправляет инцидент на проверку.

Основные действия работника:

- Вход или регистрация через APK.
- Отправка инцидента.
- Описание проблемы.
- Указание категории, даты и фактического ущерба.
- Передача инцидента риск-менеджеру.

## 4. Общий сценарий работы

1. Работник обнаруживает событие или проблему и создает инцидент в APK.
2. Инцидент появляется в очереди инцидентов в web-панели риск-менеджера.
3. Риск-менеджер проверяет инцидент и нажимает "Создать риск".
4. На основе инцидента создается риск в статусе "Создан".
5. Риск-менеджер назначает эксперта.
6. После назначения эксперта риск можно перевести в статус "В работе".
7. Эксперт видит риск в desktop-приложении.
8. Эксперт открывает риск, указывает вероятность, влияние и рекомендацию.
9. Оценка сохраняется в системе.
10. Риск-менеджер анализирует оценку и управляет дальнейшим статусом риска.

## 7. Архитектура системы

Система построена вокруг Flask backend, который предоставляет web-интерфейс и JSON API. Web, desktop и APK используют общий backend, но получают данные через разные endpoint'ы и разные DTO.

Общий принцип:

- Web-клиент используется риск-менеджером и администратором.
- Desktop-клиент используется экспертом.
- APK-клиент используется работником.
- Backend проверяет JWT-токены и роли.
- Write-модель хранит нормализованные данные.
- Read-модель хранит денормализованное представление для быстрых запросов.
- Domain events помещаются в очередь и обновляют read-модель.

## 8. Модель данных

Основные сущности:

- `User` - пользователь системы.
- `Risk` - риск.
- `IncidentIntake` - входящий инцидент от работника.
- `Incident` - инцидент, привязанный к риску.
- `MitigationMeasure` - мера минимизации.
- `Assessment` - экспертная оценка.
- `AuditLog` - журнал аудита.
- `BackgroundTask` - демонстрационная очередь фоновых задач.
- `DomainEvent` - очередь доменных событий.
- `RiskReadModel` - денормализованная read-модель риска.
- `ApiRequestMetric` - метрики API.

Write-модель нормализована: риск, инциденты, меры, оценки и пользователи хранятся в отдельных таблицах и связаны через ключи.

Read-модель денормализована: `risk_read_model` хранит готовые данные для чтения журнала рисков и BFF-ответов.

## 9. Жизненный цикл риска

Риск проходит несколько статусов:

- `CREATED` - риск создан, но еще не отправлен в работу.
- `ACTIVE` - риск в работе, эксперт может выполнять оценку.
- `COMPLETED` - риск завершен.
- `CANCELLED` - риск отменен.

Ограничения:

- Риск нельзя перевести в работу без назначенного эксперта.
- Эксперта нельзя менять, если риск уже в работе, завершен или отменен.
- Завершенный или отмененный риск нельзя редактировать как активный.

## 10. Web-версия

Web-версия предназначена для риск-менеджера и администратора.

Основные разделы:

- Журнал рисков.
- План минимизации.
- Фоновые задачи.
- Отчеты.
- Пользователи.
- Профиль.

### 10.1 Журнал рисков

Журнал рисков показывает список рисков, владельца, эксперта, оценку, статус и доступные действия. Риск-менеджер может назначить эксперта и открыть карточку риска.

### 10.2 Очередь инцидентов

Очередь инцидентов показывает сообщения от работников из APK. Риск-менеджер проверяет инцидент и может создать из него риск.

Инцидент сам по себе не является риском. Он становится риском только после действия риск-менеджера.

### 10.3 План минимизации

Раздел нужен для создания риска и мер минимизации. Меры минимизации описывают действия, ответственных, сроки и приоритет.

### 10.4 Фоновые задачи

Раздел демонстрирует обработку системных задач и очереди событий. В проекте фоновые задачи используются для отображения идеи background processing: пересчет read-модели, обработка событий, обновление сводки и журнал аудита.

В демонстрационной версии очередь обрабатывается вручную кнопкой, а в промышленной версии этот механизм можно вынести в Celery/RQ/worker.

### 10.5 Отчеты

Раздел отчетов позволяет выгружать данные по рискам. Экспорт используется для передачи руководству или для приложения к защите проекта.

### 10.6 Пользователи

Администратор видит заявки пользователей и может одобрять учетные записи. Это защищает систему от случайной регистрации и доступа к данным.

## 11. Desktop-приложение эксперта

Desktop-приложение предназначено для экспертов.

Основные возможности:

- Авторизация эксперта.
- Запоминание пользователя.
- Просмотр назначенных рисков.
- Фильтрация рисков.
- Открытие карточки риска.
- Отправка оценки.
- Просмотр своих оценок.
- Просмотр профиля.

Desktop-клиент подключается к web/API и получает только данные, доступные роли эксперта. Риск-менеджер не должен выполнять экспертные функции через desktop-приложение.

## 12. APK-приложение работника

APK-приложение предназначено для работника.

Основные возможности:

- Вход в аккаунт.
- Регистрация с заявкой на одобрение администратором.
- Настройка адреса сервера.
- Отправка инцидента.
- Указание описания, категории, даты и ущерба.

Работник не создает риск напрямую. Он отправляет инцидент, который появляется в web-панели риск-менеджера.

## 14. CQRS

В проекте реализовано разделение команд и запросов.

Команды изменяют данные и работают через ORM:

- Создание риска.
- Изменение статуса риска.
- Назначение эксперта.
- Обработка событий.

Запросы читают денормализованную read-модель и используют raw SQL:

- Получение списка рисков.
- Получение карточки риска.
- Проверка задержки read-модели.

Такое разделение упрощает масштабирование и показывает отличие write-модели от read-модели.

## 15. Backend For Frontend

BFF endpoint'ы возвращают данные, подготовленные под конкретный интерфейс.

Для web:

- сводка;
- карточки рисков;
- журнал аудита.

Для APK:

- профиль работника;
- очередь его инцидентов.

Для desktop:

- профиль эксперта;
- назначенные риски;
- последние оценки.

Это позволяет не заставлять каждый клиент собирать данные из большого количества endpoint'ов.

## 16. Domain Events, очередь и eventual consistency

В проекте есть доменные события:

- `RiskCreated`
- `RiskStatusChanged`
- `RiskExpertAssigned`
- `RiskAssessed`
- `IncidentIntakeRegistered`
- `RiskCreatedFromIncidentIntake`

События сохраняются в таблицу `domain_event`. После выполнения команды read-модель обновляется не сразу, а через обработчик очереди. Это демонстрирует eventual consistency.

Проверка задержки:

```text
GET /api/v1/queries/lag
```

Обработка очереди:

```text
POST /api/v1/commands/events/process
```

## 18. Метрики и тепловая карта

Система записывает метрики API-запросов в таблицу `api_request_metric`.

Endpoint тепловой карты:

```text
GET /api/v1/heatmap
```

Он показывает:

- самые частые query endpoint'ы;
- самые тяжелые command endpoint'ы;
- состояние очереди событий.

Кэш запросов инвалидируется после команд, чтобы read-данные не оставались устаревшими.

## 20. Минимальный комплект для переноса

Для демонстрации на другом ПК нужны:

- папка `web/`;
- собранный desktop `.exe` или `.msi`;
- собранный APK;
- Python 3.11+;
- зависимости из `web/requirements.txt`.

Не обязательно переносить:

- `node_modules`;
- `.venv`;
- исходники `apk/`, если APK уже собран;
- исходники `desktop-tauri/`, если desktop уже собран.

Если нужна текущая база с пользователями и рисками, надо перенести `web/instance/`.

Если нужна чистая база, можно не переносить `web/instance/` или выполнить сброс демо-БД.

## 21. Проверка работоспособности перед сдачей

Даже если раньше все работало, перед сдачей стоит выполнить короткую проверку. Полное тестирование заново не обязательно, но smoke-test нужен, потому что на другом ПК могут отличаться IP, firewall, зависимости и база.

Минимальный чеклист:

1. Запустить web/API.
2. Открыть `http://127.0.0.1:5000`.
3. Войти администратором или риск-менеджером.
4. Проверить журнал рисков.
5. Проверить `http://127.0.0.1:5000/api/ping`.
6. Запустить desktop и войти экспертом.
7. Проверить, что назначенный риск отображается.
8. Запустить APK или эмулятор.
9. Проверить подключение к `http://<IP_ПК>:5000/api/ping`.
10. Отправить тестовый инцидент из APK.
11. Убедиться, что инцидент появился в web.
12. Создать риск из инцидента.
13. Назначить эксперта и перевести риск в работу.
14. Проверить, что риск появился в desktop.
15. Отправить экспертную оценку.
16. Проверить отчет и журнал аудита.

- `feature/*` - отдельные функции.

Примеры веток:

- `feature/cqrs-commands`
- `feature/bff-mobile`
- `feature/desktop-expert`

Примеры коммитов:

- `feat: add incident intake flow`
- `fix: prevent expert reassignment for active risks`
- `refactor: split query and command api`
- `docs: update project documentation`

Pull Request должен содержать:

- что изменено;
- зачем изменено;
- как проверить;
- скриншоты, если менялся UI.

## 23. Соответствие требованиям задания

### 23.1 CQRS

Выполнено:

- command endpoint'ы отделены от query endpoint'ов;
- write-модель нормализована;
- read-модель денормализована;
- queries используют raw SQL;
- commands используют ORM.

### 23.2 Backend For Frontend

Выполнено:

- отдельный BFF для web;
- отдельный BFF для mobile;
- отдельный BFF для desktop;
- разные DTO под разные интерфейсы;
- BFF агрегирует данные.

### 23.3 Events + Queue

Выполнено:

- реализованы domain events;
- есть таблица очереди событий;
- есть обработчик событий;
- read-модель обновляется через очередь.

### 23.4 Eventual Consistency

Выполнено:

- command выполняется сразу;
- read-модель обновляется с задержкой через очередь;
- задержку можно проверить через query endpoint.

### 23.5 Git

Выполнено:

- используется git-репозиторий;
- коммиты оформляются осмысленными сообщениями;
- добавлен шаблон Pull Request;
- проект загружен в GitHub.

### 23.6 Метрики и тепловая карта

Выполнено:

- метрики API пишутся в таблицу;
- есть endpoint тепловой карты;
- отображаются частые query, тяжелые command и состояние очереди.

## 25. Заключение

RiskGuard реализует связку web, desktop и APK-клиента для управления операционными рисками. В проекте есть разграничение ролей, регистрация с одобрением, обработка инцидентов, назначение экспертов, экспертные оценки, меры минимизации, отчеты, журнал аудита и фоновые задачи.

С архитектурной стороны проект демонстрирует CQRS, разные модели данных, BFF для разных клиентов, domain events, очередь событий, eventual consistency, метрики, тепловую карту и документированный API.

Проект готов к демонстрации при условии короткой проверки запуска web/API, desktop и APK на том ПК, где будет проходить защита.
